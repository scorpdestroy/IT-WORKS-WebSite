# План создания корпоративного сайта и лендинга Pyrus для компании «АЙТИ ВОРКС»

**Формат реализации:** полный контроль кода + развёртывание на собственной VPS (без WordPress/конструкторов/платформ, где код/хостинг «заперт»).
**ИИ используем как ускоритель** для генерации макетов, компонентов и черновиков контента, а затем **доводим вручную**.

---

## 1) Цели и принципы
### Цели
1) **Центральный сайт компании**: кто мы, чем занимаемся, доверие, контакты, входящие заявки.
2) **Набор лендингов под конкретные услуги** (в рассылки/рекламу). Первый — **«Внедрение/настройка Pyrus»**.
3) **SEO сразу “с фундамента”**: быстрый, структурированный, контент‑ориентированный, с понятной тематикой и перелинковкой.

### Принципы (важно для маленькой компании)
- Не «много страниц обо всём», а **мало страниц, но сильных**.
- Максимум доверия: ясные формулировки, прозрачность, реальная экспертиза.
- Техническая база: скорость, доступность, правильные метаданные, семантика, схема.

## 2) Архитектура сайта: структура страниц
### 2.1 Корпоративный сайт (минимально жизнеспособная версия)
**/ (Главная)**
- 1 экран: позиционирование (финансовый консалтинг) + CTA «Обсудить задачу»
- Коротко: 3–5 преимуществ (скорость, точность, ответственность, конфиденциальность)
- Блок «Основные направления» (финконсалтинг — центральный; автоматизация/внедрение систем — вторично)
- Мини‑кейсы/результаты (даже 1–2 штуки, но конкретно)
- FAQ (5–8 вопросов)
- Контакты + форма

**/services (Услуги)**
- Финансовый консалтинг (главный блок)
- Автоматизация/внедрение (включая Pyrus) — как отдельный блок с переходом на лендинг

**/services/financial-consulting**
- Описание услуг: бюджетирование, управленческий учёт, финмодели, диагностика, CFO‑поддержка
- Для кого / результаты / формат работы
- CTA

**/services/pyrus (или /pyrus — как отдельный лендинг, см. ниже)**

**/cases (Кейсы)**
- 1–3 кейса, даже если небольшие: проблема → подход → результат → цифры/эффект

**/about (О компании)**
- Миссия/подход
- Основатель: опыт, компетенции
- Партнёрства/сертификаты (когда появятся)

**/blog (Блог)**
- 2–3 статьи на старте, дальше — регулярное пополнение

**/contacts**
- Контакты, форма, мессенджеры

**служебные:**
- /privacy, /terms, /cookies (по необходимости), /gdpr (если актуально), /data-processing

### 2.2 Лендинг «Внедрение Pyrus»
**/pyrus** (одностраничник или страница + несколько подстраниц)
- 1 экран: «Pyrus за N дней: настройка процессов + интеграции + обучение»
- Боли: хаос задач/согласований, ручные таблицы, потеря статусов
- Решение: Pyrus + наша настройка
- Что делаем: аудит → прототип (демо) → внедрение → интеграции → обучение → поддержка
- Интеграции: 1С/почта/CRM/Диадок/Telegram и т.д.
- Кейсы/сценарии (даже “типовой пример” с цифрами и оговоркой, если нет реального)
- FAQ
- CTA: «Запросить демо / оценку»

**Важно:** лендинг должен иметь смысл **сам по себе**, но при этом вести на корпоративный сайт (доверие) и обратно (перелинковка).

## 3) Технологический стек (полный контроль кода)
Ниже — рекомендованный «быстрый и надёжный» стек, оптимальный для SEO и дальнейшего роста.

### Вариант A (рекомендую как базовый): **Next.js + MDX/Markdown (контент в репозитории)**
- **Next.js (App Router)**: SSR/SSG, отличная производительность, удобная SEO‑разметка.
- **Tailwind CSS** + **shadcn/ui**: быстро собрать аккуратный консалтинговый дизайн.
- **MDX/Markdown для блога и кейсов**: контент правится прямо в Git.
- **Sitemap/robots/OpenGraph**: генерируем автоматически.

Плюсы: минимальная инфраструктура, удобно деплоить на VPS, полный контроль, быстрый старт.

### Вариант B (если хочется ещё проще и супер‑быстро): **Astro + Markdown**
- Отлично подходит для маркетинговых сайтов.
- Лёгкий, очень быстрый.
- Если интерактива мало — Astro может быть даже лучше.

### Контент: когда понадобится админка
Если позже захочется править контент без Git:
- **Directus** или **Strapi** (самохост на VPS) + PostgreSQL.
- Но на старте часто хватает Markdown — быстрее и дешевле.

## 4) Дизайн через ИИ → ручная реализация
Твой запрос: «сгенерировать дизайн ИИ‑сервисами и довести вручную». Вот рабочий пайплайн.

### 4.1 Быстрый пайплайн макетов
1) **Сайт‑карта и структура секций**
   - ИИ генерирует: список страниц + блоки на каждой странице.
2) **Черновые wireframes (каркас)**
   - ИИ выдаёт несколько вариантов компоновки главной и лендинга.
3) **UI‑стиль (мини‑брендгайд)**
   - Палитра (2–3 базовых + акценты), типографика, кнопки, карточки.
4) **Мокапы 2–3 ключевых экранов**
   - Главная (верх), услуги, лендинг Pyrus.

### 4.2 Какие ИИ‑инструменты использовать (без “конструкторов хостинга”)
- **Figma + AI плагины** (для вариантов блоков и текстов)
- **v0 / Lovable / подобные**: использовать **как генератор React‑компонентов**, а не как хостинг. То есть:
  - сгенерировал секцию → забрал код → положил в наш репозиторий.
- Генерация иллюстраций (если нужно): абстрактные фоны/иконки (без «людей‑стоков» лучше).

> Итог: дизайн получаем быстро, но код и инфраструктура — полностью наши.

## 5) SEO “с фундамента”: что делаем сразу
### 5.1 Техническое SEO
- Быстрый рендер: SSR/SSG, без тяжёлых библиотек.
- Core Web Vitals:
  - оптимизация изображений (next/image), lazy‑load, modern formats.
- Чёткая семантика HTML: один H1 на страницу, корректные H2/H3.
- Автогенерация:
  - **sitemap.xml**, **robots.txt**, canonical.
- OpenGraph/Twitter cards для красивых превью в мессенджерах.
- Schema.org:
  - Organization, WebSite, Service, FAQPage, BreadcrumbList.

### 5.2 Контент‑SEO (особенно важно для маленькой компании)
- Стратегия: **узкая специализация + экспертность**.
- На старте достаточно 6–10 страниц, но с сильными текстами.
- Блог: 2–3 статьи на запуск + план публикаций.

**Пример стартового контент‑пакета (минимум):**
- «Что такое управленческий учёт и как быстро навести порядок»
- «Pyrus: что автоматизировать в первую очередь (согласования/заявки/Service Desk)»
- «Как выбрать подход к внедрению: демо за 1–3 дня vs “долгий проект”»

### 5.3 Доверие (YMYL‑зона)
Для финансовой тематики поисковики любят:
- Страницу об авторе/эксперте, опыт, компетенции.
- Прозрачные дисклеймеры (что не является публичной фин‑рекомендацией и т.п.).
- Контакты, юр‑инфо (по возможности), политика обработки данных.

## 6) План разработки (пошагово, без «конструкторов»)
### Этап 1 — Подготовка
**Результаты этапа:**
- Структура страниц + список секций.
- Семантическое ядро (минимум 30–80 ключей) и карта запросов → страниц.
- Черновик текстов для Главной и /pyrus.

### Этап 2 — Дизайн (ИИ + Figma)
**Результаты:**
- Мини‑дизайн‑система (палитра, шрифты, кнопки, формы, карточки).
- Мокапы: Главная (верх+середина), /services, /pyrus.

### Этап 3 — Разработка фронта
**Результаты:**
- Репозиторий (Git) + базовая архитектура.
- Реализованы страницы:
  - /, /services, /about, /contacts
  - /pyrus
  - /blog (список) и /blog/[slug]
  - /privacy
- Общие компоненты: Header/Footer, CTA‑блоки, FAQ‑аккордеон.

### Этап 4 — Формы заявок (без внешних form‑сервисов)
Варианты:
- **Встроенный API endpoint** (Next.js route) → отправка email через SMTP.
- Логирование заявок в PostgreSQL (если нужна история).
- Спам‑защита: honeypot + rate limit + reCAPTCHA (внешняя) или self‑host friendly (hCaptcha).

### Этап 5 — SEO‑сборка и проверка качества
- Линт/форматирование.
- Проверка accessibility.
- Проверка скорости (Lighthouse).
- Проверка мета‑тегов, schema, sitemap.

### Этап 6 — Деплой на VPS
- Docker Compose или systemd + node.
- HTTPS (Let’s Encrypt).
- Логи, мониторинг, бэкапы.

## 7) Инфраструктура на VPS (рекомендованный вариант)
### 7.1 Базовая схема
- VPS (Ubuntu LTS)
- Docker + Docker Compose
- Reverse proxy: **Caddy** (проще) или Nginx
- Приложение: Next.js (node)
- (опционально) PostgreSQL, если хотим хранить лиды/контент

### 7.2 CI/CD (чтобы обновлять сайт за минуты)
Вариант 1 (простой):
- `git pull` на сервере + `docker compose up -d --build`

Вариант 2 (чуть лучше):
- GitHub Actions:
  - build image → push в registry → deploy по SSH.

### 7.3 Домены, DNS, SSL
- DNS можно держать у регистратора.
- SSL выдаёт Caddy/Nginx+certbot.
